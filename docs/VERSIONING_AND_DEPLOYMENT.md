# 版本回溯与 GitHub 上线说明

这份说明给不会编程的妹妹看。核心只要记住一句：**Git 是本地时间机器，GitHub 是云端备份和网页部署。**

## 当前保护

- `.env.local` 不会提交到 GitHub，里面的模型密钥只留在妹妹电脑。
- `node_modules/`、`dist/`、日志和浏览器截图不会提交，避免仓库又大又乱。
- 每次完成一个稳定阶段，姐姐会创建一个 Git checkpoint。以后改坏了，就能回到某个 checkpoint。
- GitHub Pages 只部署前端静态页面，适合手机、平板、电脑测试 UI 和记忆流程。
- 真正调用模型仍然建议用电脑本机开发版，因为密钥不能放到公开网页里。

## 妹妹日常怎么用

平时不用管 Git。妹妹只要让姐姐做：

- “姐姐，帮我保存一个稳定版本”
- “姐姐，回到上一个稳定版本”
- “姐姐，看看这次改了哪些”
- “姐姐，推送到 GitHub”

姐姐会处理命令、检查密钥、跑测试，再告诉妹妹结果。

## 回溯能解决什么

- 姐姐未来改错代码。
- 新功能把聊天页弄坏。
- UI 越改越丑，想回到某个版本。
- 记忆系统重构失败，想撤回。

## 回溯不能解决什么

- 浏览器 IndexedDB 里的个人聊天数据，如果妹妹自己在页面里删除并清空了，Git 不能自动找回。
- `.env.local` 里的密钥不会进入 Git，所以 GitHub 也不会替妹妹备份密钥。
- GitHub Pages 是静态站点，没有后端代理，公开网页不能直接使用妹妹本机中转密钥。

## GitHub Pages 上线步骤

姐姐已经准备好自动部署配置：`.github/workflows/deploy-pages.yml`。

还差云端仓库这一步。当前电脑没有安装 GitHub CLI，也没有配置远程仓库，所以姐姐不能在不登录的情况下替妹妹创建 GitHub 仓库。

等妹妹登录 GitHub 后，最稳的方式是：

1. 创建一个名为 `yuri-pocket` 的仓库。
2. 把本地仓库推送到 GitHub。
3. 在 GitHub 仓库 Settings -> Pages，把 Source 设为 GitHub Actions。
4. 推送 `main` 分支后，GitHub 会自动构建并发布页面。

部署成功后，地址通常类似：

```text
https://你的GitHub用户名.github.io/yuri-pocket/
```

## 为什么上线版不能直接真聊天

因为真聊天需要密钥。公开网页如果内置密钥，别人打开网页就能偷用妹妹的额度。

所以当前设计是：

- 本机版：能走 `.env.local` 和本机中转，适合真正聊天。
- GitHub Pages 版：不带密钥，适合跨设备看界面、测布局、试记忆管理流程。

未来如果要让手机也能真聊天，有两条安全路线：

- 做一个有登录和权限控制的后端。
- 做用户自己填密钥的 BYOK 模式，并明确告诉用户风险。

现在先不上这些，是为了不把第一版架构弄复杂。
